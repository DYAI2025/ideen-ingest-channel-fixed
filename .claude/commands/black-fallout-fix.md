## Context

This pattern hit 3 times in one session: a PR landed on `main` without ci-fast (either before branch-protection was active, or before the lint job existed in its current strict form), leaving backend Python files unformatted. The very next branch — yours — inherits those files, runs `black --check src/ tests/`, and CI lint fails with `would reformat <N> files`.

The fix is mechanical (always the same), but easy to forget. This skill encodes the recipe.

## Your Task

When ci-fast lint fails with `would reformat` against a file you didn't touch, apply a focused black-format commit so your real changes don't get tangled with format-only churn.

### Steps

1. Confirm the failure shape. Run:
   ```bash
   cd backend && uv run black --check src/ tests/
   ```
   Expected output starts with `would reformat …` lines and exits non-zero.

2. Identify the unformatted files. Capture them via:
   ```bash
   cd backend && uv run black --check src/ tests/ 2>&1 \
     | grep '^would reformat' \
     | sed 's|^would reformat ||'
   ```

3. Verify these files are NOT in your current branch's working diff (i.e. they're someone else's fallout, not yours):
   ```bash
   git diff --name-only main..HEAD
   ```
   If overlap exists, ABORT — the format issue is your own work; black-format then squash with the originating commit instead.

4. Apply the format-pass:
   ```bash
   cd backend && uv run black src/ tests/
   ```

5. Run the slack/test suite (or whichever marker is relevant) to confirm tests still pass:
   ```bash
   cd backend && uv run pytest -m "not slow and not e2e and not stress and not fuzz and not load" tests/
   ```

6. Stage and commit as an isolated commit. Subject must name the originating PR/commit if known:
   ```bash
   git add backend/src backend/tests
   git commit -m "$(cat <<'EOF'
   format(backend): black-pass on <originator-PR-or-commit> fallout

   <One paragraph: who merged the unformatted code, why, and that this
   commit is pure formatting with no logic changes.>
   EOF
   )"
   ```

7. Push and re-watch CI.

### Guardrails

- Refuse to run if `git status --short` shows uncommitted non-format changes. Commit those first; black-format must be its own commit.
- If `black --check` reports 0 files would reformat after step 1, ABORT — there's nothing to fix and the lint failure is something else (likely frontend lint or tsc).
- Never use `--no-verify` on the commit. If a pre-commit hook fails on the format-pass, that's a real issue (hook ordering bug); surface to user.
- This skill is NOT for fixing your own unformatted code — for that, just run black locally before staging.
